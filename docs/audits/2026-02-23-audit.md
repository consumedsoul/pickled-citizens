## 🔴 Critical Issues (Fix Immediately)

### 1. Admin Middleware is Dead Code — Not Recognized by Next.js
**File:** `app/admin/middleware.ts`
**Lines:** 1-53
**Risk:** No server-side route protection for any `/admin/*` routes

Next.js only recognizes `middleware.ts` at the project root or in `src/`. The file at `app/admin/middleware.ts` is **never executed**. This means:
1. All admin page protection relies solely on client-side email checks (bypassable)
2. RLS at the database level still protects data access, but the admin UI is accessible to anyone
3. The cookie name `sb-access-token` (line 19) is also incorrect for Supabase's naming convention (`sb-<project-ref>-auth-token`)

**Recommended Fix:**
1. Move `app/admin/middleware.ts` to project root as `middleware.ts`
2. Add a `matcher` config to only protect `/admin/*` routes
3. Fix the Supabase cookie name to match the project's actual auth cookie
4. Alternatively, use `@supabase/ssr` for proper server-side auth in middleware

**Why:** The middleware has been assumed to provide server-side protection since it was created (2026-02-17 audit), but it has never actually run.

## 🟠 High Priority (This Week)

### 2. Stale Closure Bug in Home Page Auth Listener
**File:** `app/page.tsx`
**Lines:** 103-119
**Risk:** Auth state change callback captures initial `auth` state and never updates

The `onAuthStateChange` callback at line 103 reads `auth.userId` but `auth` is not in the `useEffect` dependency array. The callback always sees `auth = { loading: true, email: null, userId: null }`. This can cause:
- Duplicate data loading when the user signs in
- Missed state transitions when auth changes

**Recommended Fix:**
Use a ref to track the current `auth.userId`:
```typescript
const authRef = useRef(auth);
authRef.current = auth;
// Then in onAuthStateChange: authRef.current.userId
```
Or restructure to use the `session` parameter from `onAuthStateChange` directly.

### 3. Modal Component Missing Keyboard Accessibility and ARIA
**File:** `src/components/ui/Modal.tsx`
**Lines:** 1-31
**Risk:** WCAG 2.1 violations; screen readers cannot identify the dialog

The new Modal component is missing:
- `role="dialog"` and `aria-modal="true"` attributes
- `aria-labelledby` pointing to the title
- Escape key handler to close the modal
- Focus trap (Tab can escape to elements behind the backdrop)
- Body scroll lock (page scrolls behind the modal)

**Recommended Fix:**
Add `onKeyDown` handler for Escape, ARIA attributes, `useEffect` to lock body scroll and manage focus. Consider using `@headlessui/react` Dialog or implementing a custom focus trap.

**Why:** The Modal is used for destructive actions (delete league, delete account). Users navigating by keyboard or screen reader cannot properly interact with it.

### 4. Supabase Client Not Typed with `Database` Generic
**File:** `src/lib/supabaseClient.ts`
**Lines:** 18, 35
**Risk:** All Supabase queries return implicitly typed data; end-to-end type safety is lost

The `createClient` calls do not pass the `Database` generic:
```typescript
// Current:
createClient(supabaseUrl, supabaseAnonKey)
// Should be:
createClient<Database>(supabaseUrl, supabaseAnonKey)
```

This means every `.from('table').select()` returns untyped data, and type safety is only achieved through manual casting after the fact (fragile and error-prone).

**Recommended Fix:**
```typescript
import type { Database } from '@/types/database';
export const supabase = createClient<Database>(supabaseUrl, supabaseAnonKey);
export const supabaseServiceRole = createClient<Database>(supabaseUrl, supabaseServiceRoleKey, { ... });
```

**Why:** This single change would make all 50+ Supabase queries across the codebase type-safe automatically.
